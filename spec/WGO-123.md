# WGO-123 — per-repo/path Jira project rules

## Summary

Add `[[jira.project_rules]]` to `~/.wgo/config.toml` so that `wgo add --jira` resolves the Jira project key from the current repo or directory rather than a single global default.

## Problem / Motivation

`jira.default_project` is one value for the whole system. Developers typically work across multiple GitHub orgs and projects — each mapped to a different Jira project — and increasingly in monorepos where different subdirectories belong to different Jira projects. Today every `wgo add --jira` call requires an explicit `--jira-project` flag unless the user happens to only ever work in one project.

## Proposed Solution

An ordered list of rules under `[jira]` in config. Each rule has a `repo` glob (matched against `owner/repo`) and/or a `path` substring (matched against CWD). When both fields are set the rule fires if either matches (OR / union). First matching rule wins; the global `default_project` / `default_type` are the fallback.

```toml
[jira]
default_project = "DSPX"
default_type    = "Task"

[[jira.project_rules]]
repo    = "virtru/*"        # matches any repo in the virtru org
project = "DSPX"
type    = "Story"           # optional: overrides default_type for this rule

[[jira.project_rules]]
repo    = "virtru/monorepo"  # fires if in this repo OR in this path
path    = "packages/billing"
project = "BILL"
```

## Inputs / Outputs / Contracts

### `internal/config/config.go`

New struct:
```go
type JiraProjectRule struct {
    Repo    string `mapstructure:"repo"`    // filepath.Match glob vs "owner/repo"
    Path    string `mapstructure:"path"`    // strings.Contains vs CWD
    Project string `mapstructure:"project"`
    Type    string `mapstructure:"type"`    // optional issue-type override
}
```

`JiraConfig` gains:
```go
ProjectRules []JiraProjectRule `mapstructure:"project_rules"`
```

New method:
```go
// ResolveProject returns (project, issueType) for the given repo and working dir.
// Rules are evaluated in order; a rule matches when its repo glob matches ownerRepo
// OR its path substring appears in cwd. Falls back to DefaultProject / DefaultType.
func (c *JiraConfig) ResolveProject(ownerRepo, cwd string) (project, issueType string)
```

`filepath.Match` is used for `repo` glob — no new dependencies.

### `internal/cmd/add.go` — `runAddWithJiraCreate`

Before calling `jira.CreateIssue`, resolve `ownerRepo`:
1. If `-r` flags were provided, use `addRepos[0]`.
2. Otherwise call `detectCurrentRepo(gitClient)` (tolerates failure — empty string disables repo-based rules, path rules still apply).

Then call `cfg.Jira.ResolveProject(ownerRepo, cwd)` and let `--jira-project` / `--jira-type` flags override the result.

## Edge Cases & Constraints

- A rule with neither `repo` nor `path` set is silently skipped.
- `filepath.Match` returns an error only for malformed patterns (e.g. `[z-a]`); a bad pattern logs a warning and is skipped rather than aborting.
- `ownerRepo` empty string (e.g. not in a git repo and no `-r` given) — repo-based rules never match; path rules still work.
- Rules are evaluated in TOML declaration order — first match wins. Users control priority by ordering rules.
- `type` override in a rule only applies when the rule matches; non-matching rules do not affect the type resolution.

## Out of Scope

- `**` double-glob patterns (`filepath.Match` does not support them).
- Per-rule `description` or `assignee` defaults.
- Auto-detecting the project from the ticket prefix in branch names (that's a separate lookup).

## Acceptance Criteria

- [ ] `[[jira.project_rules]]` entries are parsed from `~/.wgo/config.toml` without error.
- [ ] A rule with only `repo = "virtru/*"` matches `virtru/platform` and `virtru/wgo` but not `personal/hobby`.
- [ ] A rule with only `path = "packages/auth"` matches when CWD contains that substring.
- [ ] A rule with both `repo` and `path` set matches when either condition is true.
- [ ] First matching rule's `project` and `type` are used; subsequent rules are ignored.
- [ ] When no rule matches, `default_project` and `default_type` are used.
- [ ] `--jira-project` and `--jira-type` flags still override all rule-resolved values.
- [ ] A malformed `repo` glob pattern logs a warning and is skipped, not fatal.
- [ ] `go test ./internal/config/...` passes including rule-matching unit tests.
- [ ] `go build ./cmd/wgo` succeeds.

# WGO-132 — Cross-invocation on-disk cache for PR / CI status

## Summary

Add a shared on-disk cache under `~/.wgo/cache/` for GitHub PR and CI status,
keyed by repo + branch with a TTL, plus a background-refresh path. This lets
short-lived `wgo` invocations (`wgo .`, `wgo statusline`, `wgo status`,
`wgo pr`) reuse recently-fetched GitHub data instead of each re-hitting the
network, which is the actual latency fix that makes a statusline viable.

## Problem / Motivation

`internal/github/transport.go` implements an ETag + TTL cache
(`defaultCacheTTL = 60s`, `github.go:321`), but it lives **inside the HTTP
client of a single process**. Every `wgo` invocation is a fresh process, so:

- The in-memory cache starts empty every time.
- A statusline that renders once per prompt makes a fresh GitHub request every
  render (best case a conditional 304 round trip, worst case a full fetch).

This is precisely why the current Claude Code statusline had to implement its
own background-refreshed file cache in shell. That caching is generic and
belongs in `wgo`, shared across all read commands.

## Proposed Solution

### 1. Cache location and format

Under `~/.wgo/cache/` (already referenced in CLAUDE.md's storage model), store
per-repo-per-branch entries, e.g. `~/.wgo/cache/pr/<slug>/<branch>.json`,
containing the resolved `PRRef`(s) plus a `fetched_at` timestamp. Use a
sanitized filename for branches containing `/`.

### 2. Read-through with TTL

Add `internal/github` (or a new `internal/prcache`) helper:

```go
func (c *Client) PRStatus(repoPath, branch string, opts CacheOpts) ([]PRRef, CacheState, error)
```

- Fresh (age < TTL): return cached value, `CacheState = Fresh`, no network.
- Stale (age >= TTL): return cached value immediately as `Stale`, and — if
  `opts.RefreshStale` — trigger a background refresh (see §3).
- Miss: return empty + `Miss`; background refresh if requested.

Default TTL configurable via `config.toml` (`cache.pr_ttl`, default 120s).

### 3. Background refresh

A `wgo` subprocess (or a detached goroutine that outlives stdout via a small
`wgo _refresh-pr <slug> <branch>` internal command) performs the network fetch
and atomically rewrites the cache file (write temp + rename). The hot-path
command returns without waiting. A lock/lease file prevents duplicate
concurrent refreshes for the same key.

### 4. Wire into existing commands

`wgo .` and `wgo statusline` read through this cache. `wgo status` / `wgo pr`
can pass `opts.RefreshStale=true` and, for `pr`, optionally
`opts.Synchronous=true` to force fresh data.

### 5. Invalidation

`wgo sync` and PR-mutating commands (`ClosePR`, etc.) invalidate the affected
cache entry so stale review/merge state doesn't linger.

## Acceptance Criteria

- [ ] PR status persists under `~/.wgo/cache/` across separate `wgo` processes.
- [ ] A second `wgo statusline` within the TTL performs **no** network request
      (verified with a fake transport counting calls).
- [ ] Stale entries are served immediately and refreshed in the background;
      the hot path never blocks on the network.
- [ ] Concurrent invocations for the same key trigger at most one refresh
      (lease/lock test).
- [ ] Cache writes are atomic (temp + rename); a killed refresh never leaves a
      truncated JSON file.
- [ ] TTL is configurable via `config.toml`; default 120s.
- [ ] `wgo pr --refresh` (or equivalent) bypasses the cache for fresh data.

## Out of Scope

- What PR fields are cached (schema comes from WGO-133); this spec caches
  whatever `PRRef` contains.
- A long-lived daemon. This spec uses per-invocation background subprocesses;
  a persistent `wgo daemon`/`--serve` is a possible follow-up.

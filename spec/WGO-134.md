# WGO-134 — Agent session and live Jira status in work context

## Summary

Surface two context signals that `wgo`'s product vision promises but the
context does not yet expose: (1) which AI agent, if any, is actively working in
the current workspace, and (2) the live Jira status of the branch's ticket
(e.g. `In Review`). Both flow into `models.Context` so `wgo .` and
`wgo statusline` can display `🤖 claude` and `[DSPX-3241 In Review]`.

## Problem / Motivation

CLAUDE.md lists agent tracking as a headline differentiator ("record which AI
tool is working on which workspace/bookmark via `wgo agent status`") and Jira
context as part of spec-driven flow. In practice:

- There is **no `agent` command** wired into the CLI and no agent field in the
  context, so a statusline cannot indicate that an agent holds a workspace —
  a marquee use case for the AI-augmented workflow `wgo` targets.
- The context exposes the **ticket id** (WGO-130) but not the ticket's live
  Jira status. `wgo` already talks to Jira in `internal/cmd/standup.go` and
  `internal/cmd/spec_jira.go`, so the data is reachable; it just is not part of
  `wgo .`.

## Proposed Solution

### 1. Agent session tracking

- Record agent sessions in `~/.wgo/state.json` (or `~/.wgo/agents/`): tuples of
  `{workspace_root, agent_name, pid|start, last_seen}`.
- Provide `wgo agent start <name>` / `wgo agent stop` / `wgo agent status` and
  an env-based auto-detect (e.g. honor `CLAUDECODE`/`CLAUDE_*` env so Claude
  Code sessions self-register).
- `models.Context` gains `Agent *AgentRef` (`{name, since}`); populated by
  matching the current workspace root against active sessions, with staleness
  expiry.

### 2. Live Jira status

- Add a cached Jira lookup (reusing the existing Jira access path) keyed by
  ticket, stored via the WGO-132 cache with its own TTL
  (`cache.jira_ttl`, default 10m — Jira status changes slowly).
- `models.Context` `Spec`/ticket area gains `JiraStatus` and optionally
  `JiraAssignee`.
- Hot path never blocks: cache-miss omits the status and refreshes in
  background, identical to the PR-cache pattern.

### 3. Renderers

- `wgo .` text: `spec:` / ticket line shows Jira status; a new line shows the
  active agent when present.
- `wgo statusline` default template: `[DSPX-3241 In Review]` and `🤖 claude`.

## Acceptance Criteria

- [ ] `wgo agent start/stop/status` exist and persist sessions under `~/.wgo`.
- [ ] Claude Code sessions auto-register via env detection (documented).
- [ ] `models.Context.Agent` is populated for the workspace holding an active,
      non-stale session and nil otherwise.
- [ ] Ticket branches expose `JiraStatus` in `wgo . --json`, served through a
      cache with a configurable TTL (default 10m).
- [ ] Jira and agent lookups add **no** synchronous network call to the
      `wgo statusline` hot path.
- [ ] Text and statusline renderers show agent and Jira status when available
      and degrade cleanly when not (no `gh`/Jira/token → fields simply absent).

## Out of Scope

- Multi-agent conflict resolution or locking a workspace to one agent.
- Writing back to Jira (status transitions) — read-only here.
- Cross-repo effort correlation (separate concern).

## Implementation Notes (pre-start, 2026-07-10)

Prereqs satisfied: **WGO-130** (typed `models.Context`, text/JSON parity) and
**WGO-132** (cross-invocation on-disk cache) are merged in-stack. Work proceeds
on `WGO-134-agent-jira`, stacked on `WGO-133-pr-data`.

Reuse these concrete patterns so the Jira lookup meets the "no synchronous call
on the statusline hot path" AC (WGO-133 is a worked example of exactly this):

- **Cache**: mirror `internal/prcache` (its `Read`/`Write`/`Resolve` with
  `Fresh`/`Stale`/`Miss` + detached background refresh) for a Jira-status cache
  keyed by ticket, TTL `cache.jira_ttl` (default 10m). `prTTL()` in
  `internal/cmd/prsource.go` shows the config-with-fallback shape.
- **Wiring**: follow `ghFetcher` / `resolvePRs` / `cacheOpts` in
  `internal/cmd/prsource.go` — enrich off the hot path via a `Fetcher`, project
  onto a `models.Context` field, and read cache-only in `LocalOnly` mode.
- **Jira access** already exists: `internal/cmd/standup.go`,
  `internal/cmd/spec_jira.go`, and the `internal/jira` client.
- **Renderers**: extend `renderText` (`internal/cmd/dot.go`) and
  `renderStatuslineLine` (`internal/cmd/statusline_render.go`); both omit empty
  segments already, giving the required graceful degradation for free.

Agent tracking is greenfield — no `agent` command or state schema exists yet;
`internal/store` (`~/.wgo`) is the home for the session records.

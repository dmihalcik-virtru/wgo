# WGO-130 — Unified typed work `Context` with text/JSON render parity

## Summary

Refactor `wgo .` so it computes a single strongly-typed `models.Context`
value and renders both the human (text) and machine (`--json`) outputs from
that one value. Today the two outputs are produced by separate code paths and
have drifted: the text path resolves ticket, spec, and per-branch tasks, while
the `--json` path emits a hand-built `map[string]interface{}` that omits them.
This spec makes the JSON a first-class, versioned projection of the same data.

## Problem / Motivation

`internal/cmd/dot.go:showContext()` computes rich context for the terminal:

- ticket id via `spec.ParseTicketFromBranch` (dot.go:200)
- spec path + status + updated date via `spec.FindByTicket` / `spec.Parse` (dot.go:201-211)
- per-branch tasks via `plan.GetTasksForBranch` (dot.go:187)
- workspace siblings via `showSiblings` (dot.go:214)

But the `--json` branch (dot.go:131-164) builds an independent map containing
only `repo`, `branch`, `status`, `dirty`, `ahead`, `behind`, `remote`, urls,
`commit`, and (conditionally) `prs`. Ticket, spec, and tasks never appear.

This drift is not hypothetical — it actively broke a downstream consumer. A
statusline integration built against `wgo . --json` could not read the ticket,
spec state, or plan annotation even though `wgo .` prints all three, and had to
re-derive them with brittle shell (regex over the branch name, `grep` over
`~/.wgo/plan.md`). Two code paths producing "the same" context is a standing
source of this class of bug.

## Proposed Solution

### 1. A typed context model

Add `models.Context` (in `models/`) capturing everything `showContext`
resolves:

```go
type Context struct {
    SchemaVersion int          `json:"schema_version"`
    Repo          string       `json:"repo"`
    RepoURL       string       `json:"repo_url"`
    Branch        string       `json:"branch"`
    Status        string       `json:"status"` // clean|modified|staged|conflict
    Dirty         bool         `json:"dirty"`
    Ahead         int          `json:"ahead"`
    Behind        int          `json:"behind"`
    SyncUnknown   bool         `json:"sync_unknown,omitempty"` // ahead/behind lookup failed
    Remote        string       `json:"remote"`
    BranchURL     string       `json:"branch_url"`
    Commit        CommitInfo   `json:"commit"`
    Ticket        string       `json:"ticket,omitempty"`
    Spec          *SpecRef     `json:"spec,omitempty"`
    SpecMissing    bool        `json:"spec_missing,omitempty"`    // ticket present, no spec file
    SpecUnreadable bool        `json:"spec_unreadable,omitempty"` // spec file present but unparseable
    Tasks         []TaskRef    `json:"tasks,omitempty"`
    PRs           []PRRef      `json:"prs,omitempty"`
    Siblings      []SiblingRef `json:"siblings,omitempty"`
}
```

`SpecRef` carries `{path, status, updated}`; `TaskRef` carries
`{bullet, text}`; `PRRef` carries the fields already emitted plus room for
WGO-133 additions; `SiblingRef` carries `{name, branch, status_summary}` — the
sibling status field is keyed `status_summary` (a human phrase like
"2 modified, 1 added") to avoid colliding with the top-level `status` enum's
value space.

The `spec` line has three resolved states, not two: a parseable spec populates
`Spec`; an absent spec sets `SpecMissing`; a spec file that exists but fails to
parse sets `SpecUnreadable` (so a malformed spec surfaces a distinct warning
instead of silently vanishing). Spec lookup is rooted at the workspace root, so
`wgo .` resolves the spec even when run from a subdirectory. Likewise
`SyncUnknown` distinguishes a genuine ahead/behind lookup failure from a real
`↑0 ↓0`.

### 2. One resolver, two renderers

Split `showContext` into:

- `buildContext(cwd) (*models.Context, error)` — all resolution, no I/O to
  stdout. Every field the text path shows must be populated here.
- `renderText(*models.Context, tty)` and `renderJSON(*models.Context)` —
  pure projections. `renderJSON` replaces the hand-built map with
  `json.NewEncoder` over the struct.

`--porcelain` and `--exit-code` continue to read from the same context value.

### 3. Schema versioning

`SchemaVersion` starts at `1`. Additive fields do not bump it; a breaking
change (rename/removal/semantic change) bumps it and is noted in the spec
changelog. Consumers can branch on it.

## Acceptance Criteria

- [ ] `models.Context` exists and is the single value rendered by `wgo .`.
- [ ] `wgo . --json` includes `ticket`, `spec`, `tasks`, and `siblings`
      whenever `wgo .` (text) would display them, and omits them otherwise.
- [ ] `wgo . --json` emits `schema_version: 1`.
- [ ] Text output is byte-for-byte unchanged from current behavior (golden
      test) except where a field was previously missing.
- [ ] A table test asserts that for a fixture repo, every field shown in the
      text output has a corresponding populated field in the JSON output
      (parity guard against future drift).
- [ ] No `map[string]interface{}` remains in `dot.go`.

## Out of Scope

- The dedicated `wgo statusline` command (WGO-131).
- On-disk PR caching (WGO-132).
- New PR fields such as review decision / CI (WGO-133).
- Agent / live Jira enrichment (WGO-134).

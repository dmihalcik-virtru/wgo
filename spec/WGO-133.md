# WGO-133 — Richer PR data: review decision, draft, and CI rollup

## Summary

Extend `PRInfo` (and therefore the `models.Context` PR entries) with the
review decision, draft flag, and a CI/checks rollup. These are the highest-value
glanceable PR signals and are entirely absent today, forcing consumers to make
a second `gh` call to learn whether a PR is approved or its checks are green.

## Problem / Motivation

`internal/github/github.go:238` defines `PRInfo` with `Number`, `State`,
`Branch`, `BaseRefName`, `HeadSHA`, `MergeCommit`, `MergedAt`, `ClosedAt`,
`URL`, `Title`, `Author`. It has:

- no **review decision** (approved / changes requested / review required), and
- no **CI/checks** status.

So `wgo .` can say a PR exists and whether it is open/merged/closed, but not
whether it is approved or passing CI. The Claude Code statusline had to shell
out to `gh pr view --json reviewDecision,isDraft` separately to show a `✓`/`✗`
marker — data `wgo` should already own. CI status in particular is arguably the
single most useful at-a-glance signal for someone juggling many branches.

## Proposed Solution

### 1. Extend `PRInfo`

```go
ReviewDecision string    `json:"reviewDecision"` // APPROVED | CHANGES_REQUESTED | REVIEW_REQUIRED | ""
IsDraft        bool      `json:"isDraft"`
Checks         CIStatus  `json:"checks"`
```

```go
type CIStatus struct {
    State   string `json:"state"` // success | failure | pending | none
    Passed  int    `json:"passed"`
    Failed  int    `json:"failed"`
    Pending int    `json:"pending"`
    Total   int    `json:"total"`
}
```

### 2. Populate from the GitHub API

- `reviewDecision` and `isDraft` come from the pull request object
  (`/repos/{slug}/pulls?...` already used in `ListPRsForBranch`, plus the fields
  requested).
- CI rollup comes from the combined status / check-runs API for `HeadSHA`
  (`/repos/{slug}/commits/{sha}/check-runs` or the Statuses API), summarized
  into `CIStatus`. Fetch lazily and fold into the same cache (WGO-132) so it is
  not an extra hot-path call.

### 3. Surface in context + renderers

- `models.Context` `PRRef` gains `ReviewDecision`, `IsDraft`, `Checks`.
- `wgo .` text output gains a compact marker, e.g.
  `pr: #45 Title [OPEN ✓ CI:green]`.
- `wgo statusline` default template can render `#45✓` (approved) / `✗`
  (changes) / `•` (draft) and a CI dot.

## Acceptance Criteria

- [ ] `PRInfo` exposes `ReviewDecision`, `IsDraft`, and a `Checks` rollup.
- [ ] `wgo . --json` PR entries include these fields.
- [ ] CI rollup correctly summarizes mixed check-run results
      (success/failure/pending) into a single state (table test with fixtures).
- [ ] Review decision maps GitHub's values to the documented enum, with `""`
      when none applies.
- [ ] The additional data is fetched through the WGO-132 cache — no extra
      synchronous network call on the `wgo statusline` hot path.
- [ ] Text and statusline renderers display the new signals.

## Out of Scope

- Per-check detail / links (only a rollup is required here).
- Required-reviewers or branch-protection introspection.

## Implementation Notes (2026-07-10)

Delivered on `WGO-133-pr-data`, stacked on `WGO-132-prcache`.

**Types.** `models.CIStatus{State,Passed,Failed,Pending,Total,URL}` lives in the
dependency-free `models` package; `internal/github` imports it (never the
reverse), so `PRInfo.Checks` and `PRRef.Checks` share one type with no
conversion. `PRInfo` and `PRRef` both gained `ReviewDecision`, `IsDraft`,
`Checks`.

**Fetch.** `github.ListPRsForBranchEnriched` wraps the base one-GET
`ListPRsForBranch` and fans out per PR (parallel, bounded by `per_page=5`) for
reviews + checks. Only the prcache fetcher (`ghFetcher.FetchPRs`) calls it, so
the base list stays cheap and the enriched fields ride the WGO-132 cache — the
`wgo statusline` hot path makes no extra network call (verified by a prcache
round-trip test).

**Rollups (pure, table-tested in `internal/github/enrich_test.go`).**
- `summarizeChecks` — check-runs + legacy commit statuses; precedence
  `failure > pending > success`, `none` when empty. neutral/skipped count as
  passing; timed_out/cancelled/action_required count as failing.
- `rollupReviewDecision` — latest decisive review per user; any
  `CHANGES_REQUESTED` wins, else any `APPROVED`, else `""`.
- `ciDeepLink` — per-state links derived from the PR's own `html_url`.

**Deviations from the plan above.**
- CI markers are OSC8 hyperlinks (failing job / merge-queue `…/queue/{base}` /
  checks tab), pulling a little "per-check link" back in from Out of Scope by
  request — still a rollup, not per-check detail.
- `ReviewDecision` is computed from the REST reviews endpoint (`reviewDecision`
  is GraphQL-only). `REVIEW_REQUIRED` is **not** synthesized (it needs
  branch-protection data, out of scope); `""` is emitted instead.

**Render style.** Hybrid — `wgo .` uses words (`[OPEN ✓ CI:green]`),
`wgo statusline` uses a review glyph + colored CI dot (`#42 open ✓ ●`).

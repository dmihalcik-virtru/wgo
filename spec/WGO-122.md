# WGO-122 — `wgo add`: Jira ticket creation and spec enrichment

## Summary

Extend `wgo add` with a `--jira` flag to create a Jira work item before setting up worktrees, and silently enrich the spec scaffold with Jira data when an existing ticket is provided. Add a `--push` flag to `wgo spec jira sync` to write spec changes back to Jira.

## Problem / Motivation

`wgo add` and Jira are currently disconnected in two directions:

**Outbound (wgo → Jira):** When starting a new task, the developer must manually create a Jira ticket, copy the key, then run `wgo add TICKET "desc"`. This is two context switches and a copy-paste, and the Jira ticket ends up with a thin description because no one wants to write it twice.

**Inbound (Jira → spec):** When a Jira ticket already exists with a well-written summary and description, `wgo add WGO-XXX "desc"` ignores all of that and leaves the spec with a one-line placeholder. The developer manually re-enters information that already exists.

**Round-trip (spec → Jira):** After a spec is written and refined locally, there is no path to push the improved title or description back to Jira — the ticket stays stale.

## Proposed Solution

### 1. `wgo add --jira "desc" -r owner/repo` — create Jira ticket first

When `--jira` is passed (no `TICKET` arg given), `wgo add`:
1. Creates a Jira work item via `acli jira workitem create --summary "desc" --project PROJECT --type TYPE --json`
2. Parses the returned JSON to extract the new ticket key (e.g. `WGO-123`)
3. Proceeds identically to `wgo add WGO-123 "desc"` — worktrees, spec scaffold, plan update
4. Prints the new ticket key to stderr alongside the existing progress output

`PROJECT` and `TYPE` are resolved in order: command-line flags → `~/.wgo/config.toml [jira]` defaults → error if neither is set.

New flags on `wgo add`:
```
--jira                           Create a new Jira work item before proceeding
--jira-project PROJECT_KEY       Project key for new ticket (default: jira.default_project in config)
--jira-type   ISSUE_TYPE         Issue type for new ticket (default: jira.default_type in config, e.g. "Task")
```

### 2. Silent Jira enrichment for existing tickets

When `wgo add WGO-XXX "desc"` is run (existing ticket), silently call `jira.GetIssue(ticket)` and, if successful, use the Jira data to populate the spec template:

- Spec `title` (frontmatter + heading) ← Jira `fields.summary`
- Spec `## Summary` body ← Jira `fields.description` (ADF extracted to plain text), falling back to `desc`
- Spec `jira_priority` frontmatter field ← Jira `fields.priority.name`

If `acli` is unavailable, not authenticated, or the ticket is not found, fall back silently to using the `desc` arg as today — no error, no behavior change.

A brief note is printed to stderr when enrichment is used: `enriched spec from Jira: WGO-XXX`.

### 3. `wgo spec jira sync --push [TICKET]` — write spec changes back to Jira

Add `--push` flag to the existing `wgo spec jira sync` command. When set:
1. Reads the spec `title` and the body of the `## Summary` section
2. Calls `acli jira workitem edit --key TICKET --summary "title" --description "summary body" --yes`
3. If the spec `status` is terminal (`shipped` → `Done`, `abandoned` → `Won't Do`), also calls `acli jira workitem transition --key TICKET --status "..." --yes`

Without `--push`, `sync` continues to pull from Jira only (existing behavior).

`CreateIssue` JSON parsing: parse `{"key": "WGO-123", ...}` from stdout, return `key`.

```go

```
with a `DescriptionText() string` method (ADF extraction already implemented in `extractText`).
`GetIssue` fields list expands to include `description`.

```go
title, specDesc, priority := fetchJiraEnrichment(ticket, desc)

```

Helper:
```go
func fetchJiraEnrichment(ticket, fallback string) (title, description, priority string)
```
Calls `jira.GetIssue`; returns `(fallback, fallback, "")` on any error.

After writing spec file, if `priority != ""`:
```go
_ = spec.UpdateFrontmatter(specAbs, func(fm *spec.Frontmatter) error {
    fm.JiraPriority = priority
    return nil

```

### `internal/cmd/spec_jira.go` changes

`specJiraSyncCmd` gains a `--push` flag (bool). When set, after the existing frontmatter update:
1. Re-read spec to extract `## Summary` section body
2. Call `jira.UpdateIssue(ticket, title, summaryBody)`
3. If spec status is `shipped` or `abandoned`, call `jira.TransitionIssue(ticket, mappedStatus)`

Spec → Jira status mapping (`MapSpecStatus`):
| Spec status | Jira status |
|---|---|
| `shipped` | `Done` |
| `abandoned` | `Won't Do` |
| `in_progress` | `In Progress` |
| `draft` | `To Do` |

`MapSpecStatus` lives in `internal/jira/client.go` alongside the existing `MapJiraStatus`.

## Edge Cases & Constraints

- **`acli` not found during `--jira` create**: hard error — creating the ticket is the user's explicit intent, can't silently skip.
- **`acli` not found during silent enrichment**: silent no-op; `desc` arg used as fallback.
- **`--jira` combined with a TICKET arg**: error — the two are mutually exclusive.
- **`--jira` without `-r` and not in a git repo**: same error path as today (detectCurrentRepo).
- **`--jira-project` / `--jira-type` required resolution**: error if neither flag nor config value is available. Error message should name the missing config key.
- **Jira description is empty**: use `desc` arg for spec body even when title came from Jira.
- **`--push` transition**: only transition when spec status is terminal (`shipped`/`abandoned`); `draft` and `in_progress` do not trigger a transition call — status changes should be deliberate.
- **`UpdateIssue` description length**: truncate at 32 KB (Jira ADF limit); log a warning to stderr.
- **`--push` without Jira auth**: clear error from `CheckAuth()` called at start of push path.

## Out of Scope

- Watching Jira for changes and auto-syncing.
- Creating Jira epics or setting parent/sprint fields.
- Bulk operations (creating tickets for multiple repos simultaneously).
- Interactive prompts for project/type selection (use flags or config).
- Editing Jira comments or attachments.
- Reconciling diverged title/description (last-write-wins in push direction).
